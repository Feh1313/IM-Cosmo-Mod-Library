Idol Career Diary rendering changes

- Random event conclusion:
  - Replace the raw `reply_effect_summary` row in normal view with parsed localized effect rows.
  - Prefer a new structured IM Data Core effect field when present; otherwise parse the legacy packed summary.
  - Show only confident humanized rows in normal view. Keep raw effect data debug-only.
- Substory lifecycle:
  - Resolve the visible subject/title in this order: new optional payload display name if later available, runtime mapping for known story ids, cleaned id as last resort.
  - Hide `TextSubstoryType` in normal view unless it adds information not already conveyed by the event title.
  - Show a parent-story row only when a distinct parent can be resolved cleanly.
- Task lifecycle:
  - Add a localized task summary built from task category, goal, route, required skill, assigned idol, and agent when present.
  - Keep supporting rows only when they add information beyond that summary.
  - Do not show raw task-type or route tokens in normal view.
- Research events:
  - Keep the event titles, but rewrite the detail rows so they read as research actions/results, not raw properties.
  - Use localized rows around research area and affected parameter only.
- Rival market events:
  - Parse the legacy trend summaries into localized rising/falling lines per dimension.
  - Show separate rising and falling rows for genre, lyrics, and choreography.
  - Include points only when the runtime payload actually provides them.
  - Do not show the raw compact summaries in normal view.
- Cafe lifecycle:
  - Parse `menu_summary` into weekday -> dish output.
  - Resolve dish titles from ids when possible. If a dish id no longer resolves, fall back to a localized `Dish #{0}` format, not the raw summary blob.
  - Keep the raw summary debug-only.
- Mentorship events:
  - Parse `mentor_pairs_summary` into idol-name pairs.
  - Show `Active Mentorships` only when it adds context beyond the already-focused mentor/kohai pair.
  - Omit it on single-pair events when redundant.
- Bankruptcy and scandal checks:
  - Replace raw `triggered_dialogue` display with explicit localized outcome text for the known story flags.
  - Omit the row in normal view when the same outcome is already fully explained by the boolean transition rows.
  - Keep raw ids debug-only.
- Player date interaction:
  - Stop showing raw `date_result_token` in normal view.
  - Prefer a new normalized summary code from IM Data Core; otherwise derive a summary from the legacy token/null state.
  - Keep raw tokens debug-only.
- Tour and election detail:
  - Add explicit localized code labels for all tour-country and election-broadcast values used by the base game.
  - Keep `Country` and `Broadcast` labels unless the enum audit proves the label itself is misleading.
- Generic fallback policy:
  - `HumanizeCode` remains a last-resort fallback, but none of the touched player-facing event families should depend on it for their primary normal-view text.

Payload and Interface Changes

- Additive IM Data Core payload changes only:
  - `RandomEventConcludedEventPayload` gains optional `reply_effect_entries`, a structured array of objects with `target`, `parameter`, `formula`, and `special`.
  - `PlayerDateInteractionPayload` gains optional `date_result_summary_code`.
- Keep these legacy fields unchanged and still serialized:
  - `reply_effect_summary`
  - `date_result_token`
- Do not add IM Data Core fields for cafe menu, mentor pairs, rival trends, loan enums, task enums, or research enums in this pass. Their existing payloads are already structured enough for Idol Career Diary to parse.
- Do not rename or remove any existing JSON field name, payload member, or localization key id.
- Add named constants for every new payload field name, summary code, legacy token code, delimiter, and parser index used in edited code.
- If IM Data Core needs array serialization support for `reply_effect_entries`, add a reusable helper for object arrays and use it only for this new optional field in this pass.
- Normalize `date_result_summary_code` as:
  - `dialogue_followup` when the base-game result token list is `null`
  - `no_special_result` when the list is empty or all tokens are empty
  - `public_date` when the effective token is `pub`
  - `routine_date` when the effective token is `generic`
  - `multi_result` when multiple non-empty tokens are present
  - otherwise preserve the single normalized token as the summary code and localize it through `c.CodeLabel.*`
- Do not add a guessed substory title field in this pass. The base dialogue object does not expose a stable standalone title, so the diary must resolve or map story names at render time and fall back to cleaned ids when no trustworthy title exists.

Assumptions

- Use the existing tracked planning file as the text doc to update first once execution is allowed.
- The strict no-loose-literals/no-magic-numbers rule applies to the touched event families and edited helpers, not a whole-mod sweep.
